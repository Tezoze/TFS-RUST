# Engine Architecture & Implementation Lessons

1. **Legacy C++ Parity**: Always refer back to the exact legacy TFS C++ source code when answering questions, reproducing behaviors, or making architectural decisions for edge cases. Do not assume behavior; find the correct `.cpp` file using `grep_search` in the legacy `src/` directory and mirror its logic. 
*(Added: April 2026 after spawning parser error logic mismatch)*

2. **TFS RSA is raw modular exponentiation, not PKCS#1 decrypt**: The `rsa` crate’s `rsa::hazmat::rsa_decrypt` applies blinding and PKCS#1-style semantics for standard encrypted messages. TFS (`src/rsa.cpp`) uses CryptoPP `CalculateInverse` — effectively **c^d mod n** on the 128-byte integer with **no** PKCS#1 unpadding. Those two paths are not equivalent; using the hazmat path against OTClient ciphertext can decrypt to garbage even with the correct key. The fix is **`BigUint::modpow(&d, &n)`** (or equivalent) and then fixed-width 128-byte big-endian encoding of the result, matching C++ `Integer::Encode`. See `crates/tfs-rust-net/src/rsa.rs`.  
*(Added: April 2026 — handshake “wrong key” / PKCS#1 first-byte errors)*

3. **Login RSA offset depends on client protocol version and OTClient branch**: Do not assume a single “legacy” layout. For example **protocol 1098 / OTClient v8** `ProtocolLogin:sendLoginPacket` prepends **`ClientEnterAccount` (u8 opcode 0x01)** and fields such as **u32 client version**, **u16 content revision**, padding, **spr/pic signatures**, and **u8 preview state** before the RSA block — so the ciphertext starts at **offset 26** from the start of the frame body **after** the 4-byte Adler checksum prefix (i.e. `body[26..158]` when `body[0..4]` is checksum). Older layouts use **15** (game), **25** (login ≥971), or **20** (login before 971). When in doubt, read the **actual client source** (e.g. OTClient v8 Lua) rather than guessing offsets. See `crates/tfs-rust-net/src/game_first_packet.rs`.  
*(Added: April 2026 — RSA decrypt OK only after fixing offset for 1098)*

4. **Shutdown save vs logout save**: On normal disconnect, `PlayerSaveData` is built on the `LocalSet` game thread and passed into `tokio::spawn` (I/O off the sim loop; DB is `Send`). On **SIGINT**, the same `build_player_save_data` + `PlayerStore::save_player` must **finish before process exit** — use a dedicated `GameCommand::Shutdown` branch and **`await`** a bounded work queue (`JoinSet` with a max in-flight cap of 8 matches the plan’s `buffer_unordered(8)`). Do not block the game port `accept` loop on the main `LocalSet` future: `spawn` both listeners so `await`ing the `spawn_local` game task can run to completion, then `abort` the acceptors. *(`tasks/config-and-save-plan.md` P4, April 2026)*

5. **Config.lua vs environment (network + DB)**: `NetConfig` / `DbConfig` in `config.rs` mirror TFS C++ `configmanager.cpp` keys and defaults. For containers and one-off tests, **env still overrides**: `DATABASE_URL` replaces the `mysql://…` string built from `mysqlHost` / `mysqlUser` / …; `TFS_LOGIN_ADDR` and `TFS_GAME_ADDR` override bind addresses; `TFS_GAME_PORT` and `TFS_PUBLIC_IP` override what clients are told for the game server. If unset, values come from `config.lua` (and `TFS_CONFIG` chooses which file to load). *(`tasks/config-and-save-plan.md` P1–P2 + P5, April 2026)*

6. **Password hashing (bcrypt + legacy SHA1)**: Rust auth deviates from TFS 1.4.2 plain SHA1. Stored format is self-describing: 40 lowercase hex = legacy SHA1; `$2a$`/`$2b$`/`$2y$` = bcrypt. Successful SHA1 login upgrades to bcrypt on the spot. **Both** bcrypt verify and bcrypt hash run on `tokio::task::spawn_blocking` (I/O path, not game thread); sync `hash_bcrypt` / `verify_password_sync` are for tests and blocking callers only. `$2b$` hashes verify under PHP `password_verify`. Set `legacySha1Enabled = false` in `config.lua` once migration is complete. **Startup runs SQLx migrations** (`run_migrations` in `run_server.rs`). For existing C++ TFS databases, `heal_and_adopt_existing_schema` clears failed `_sqlx_migrations` rows and marks the baseline applied without re-running DDL; only incremental Rust migrations (e.g. password widen) execute. Baseline seed uses `INSERT IGNORE` into `server_config`. *(`crates/tfs-rust-db/src/migrations.rs`, May 2026)*

7. **Player `queryDestination` BFS order**: For `CONST_SLOT_WHEREEVER`, C++ scans equipment slots 1–10 before BFS into bags — stackable merge on a worn stack wins over an empty slot; container roots are queued only after the slot pass. Rust mirrors this in `player_query_destination_wherever` using `ContainerIterator` for deep `queryMaxCount` the same way as C++ `ContainerIterator` in `Player::queryMaxCount`. Store inbox (slot 11) is intentionally excluded (`CONST_SLOT_LAST` = ammo). *(`player_inventory_query_add.rs`, May 2026)*

8. **Walk-to-item throw pre-check (intentional deviation)**: C++ `game.cpp` ~970–983 does not pre-validate throw reachability before `getPathTo` + `setNextWalkActionTask`; the player walks to the item and may fail at execution. Rust `throw_dest_reachable_after_walk_to_item` rejects early with `DestinationOutOfReach` when no adjacent stand tile can throw to the destination — approved UX improvement; valid throws are unchanged. **`try_walk_to_and_action` empty path**: when `getPathMatching` returns an empty dir list (already within `maxTargetDist`), C++ still calls `startAutoWalk([])` → `onWalkComplete` → 400 ms `walkTask`. Returning `false` from `try_walk_to_and_action` incorrectly sends `ThereIsNoWay`; schedule via `on_player_walk_complete` instead. **`dirNeighbors` index**: C++ uses `parent - current` (direction *came from*), not travel direction — `DIRECTION_WEST` when parent is west of current. *(`3951145` walk/pathfinding port, May 2026)*

9. **`playerLookAt` uses `STACKPOS_LOOK`, not `STACKPOS_MOVE`**: C++ `Game::internalGetThing(..., STACKPOS_LOOK)` ignores client `stackPos` on map tiles and calls `Tile::getTopVisibleThing` — creature first, then first non-`lookThrough` down/top item, then ground. Rust initially wired look to `internal_get_thing_move`, which only returned moveable down items and never ground — so grass/water/trees showed cancel instead of `"You see …"`. Fix: `internal_get_thing_look` + `LookTarget::Ground(u16)` with ephemeral `Item` for `Item::getDescription`. Rust stores tile ground as `TileBody.ground: Option<u16>` (type id), not a `SlotMap` `ItemId`. Creature look text still stubbed until Lua `onLook` / `Creature::getDescription` port. *(`game_world_inventory.rs`, `tile.rs`, May 2026)*

10. **Cross-floor look uses `ProtocolGame::canSee`, not same-z viewport**: C++ allows looking at visible tiles on other floors (surface z 0–7 from any surface tile; underground ±2). Viewport bounds shift by `offsetz = viewer.z - target.z`. Rust `can_see_position` was same-floor-only, blocking look-at on floors above/below; fixed by porting `protocol_can_see` (`protocolgame.cpp` ~796). Look distance still adds +15 per floor via `look_distance_tfs`. *(May 2026)*

11. **Inventory postAdd/postRemove side effects**: C++ funnels equip/deequip, weight, light, stats, container UI, and `Player:onInventoryUpdate` through `Player::postAddNotification` / `postRemoveNotification` with `cylinderlink_t` (`LINK_OWNER` vs `LINK_TOPPARENT`). Rust centralizes in `player_inventory_notifications.rs`; slot mutations use `equip_item_to_inventory_slot` / `unequip_item_from_inventory_slot`; backpack changes use `notify_player_container_tree_changed` (TopParent). Lua inventory update requires `fire_on_player_inventory_update` (ScriptContext + mutation scope) — not a bare `EventDispatcher` call. Trade `checkTradeState`, shop `updateSaleShopList`, and depot-owner container branches remain stubbed. MoveEvent ability application still deferred; `inventory_abilities[]` clears on deequip so double-apply is prevented when MoveEvents land. *(`player_inventory_notifications.rs`, May 2026)*

12. **P3 inventory utilities scan scope**: `getItemTypeCount` / `removeItemOfType` iterate `CONST_SLOT_FIRST`..=`CONST_SLOT_LAST` (slots 1–10) only — store inbox (11) excluded, same as `queryDestination`. `removeItemOfType` uses C++ if/else-if: equipped direct match when `!ignoreEquipped`, else container `ContainerIterator` only (no descent after a slot-root match). `Item::countByType` uses ported `get_sub_type` (fluid / stack count / charges). Batch removal reuses `container_remove_thing` and `unequip_item_from_inventory_slot` for notifications. `getWeaponSkill` uses raw `Player.skills.*` until full `getSkillLevel` modifiers exist. GM capacity via `groups.xml` → `player_flags.rs` on `GameWorld.groups`. *(`player_inventory_util.rs`, `game_world_inventory.rs`, May 2026)*

13. **P5 Lua inventory API boundaries**: Reads extend `ScriptContext` in `tfs-rust-common` (`ScriptCylinder`, container queries, `find_player_item_by_type`); mutations extend `LuaMutation` with immediate `apply_lua_mutation` in `lua_scope.rs` so scripts see mid-callback state. `ContainerRef` shares `ItemId` with `ItemRef`; mlua metatable `__index` chains Container → Item for compat with `data/lib/compat/compat.lua`. Equip MoveEvents run under `with_equip_mutation_scope` because callbacks may call `moveTo`/`addItem`. `player_query_add` equip probe requires `&mut GameWorld` — propagate `&mut self` through query chain or snapshot item fields before the probe to satisfy borrow checker. MoveEvents loader injects default `onEquipItem`/`onDeEquipItem` when `movements/lib/movements.lua` is empty; `req_level` checked in Rust before Lua. Deferred: full `data/lib` load, XML `inventoryAbilities`, compat `MoveEvent():register()`. *(`tfs-rust-lua`, `lua_scope.rs`, `move_events.rs`, May 2026)*

14. **`is_map_tile` must not test `pos.y & 0x40`**: C++ `Game::internalGetCylinder` treats any `pos.x != 0xFFFF` as a map tile (`game.cpp` ~199). The `0x40` bit in `pos.y` is **only** the open-container encoding when `pos.x == 0xFFFF`. Rust incorrectly required `(pos.y & 0x40) == 0` for map use — tiles with y ≥ 64 (e.g. y=126) were skipped for ladder/grate teleport and fell through to container open with no visible effect. Fix: `is_map_tile = pos.x != 0xFFFF`. *(`container_ui.rs`, May 2026)*

15. **Map `AddCreature` light: `access_player` is GM-only, not `is_self`**: C++ `ProtocolGame::AddCreature` writes light level `0xFF` only when the **viewing** player `isAccessPlayer()` (`protocolgame.cpp` ~3207), and otherwise uses `creature->getCreatureLight()`. Rust wrongly set `access_player: is_self`, so every post-teleport `0x64` map refresh encoded the local player at light `0xFF` (full brightness). Unequipping a torch “fixed” it by sending a separate `0x8D` creature-light packet. Fix: pass `items_light` + `viewer_is_access_player(group.access)`. *(`login_out.rs`, May 2026)*

16. **Phase D.1 walk engine generalization**: C++ `Game::internalMoveCreature(Creature*, Direction)` applies height-based floor change **only when `creature->getPlayer()`** (`game.cpp` ~805); monsters/NPCs use flat `getNextPosition`. Walk timing (`addEventWalk`, `onWalk`, `checkCreatureWalk`) lives on base `Creature` — Rust routes `next_walk_check` / `walk_queue` / `last_step*` through `CreatureBase`. `Player::getStepSpeed` clamps to `PLAYER_MIN/MAX_SPEED`; base creatures use raw `getSpeed()`. Spectator `0x6D` for non-players uses `non_player_wire_id(cid)` (slot key low 32 bits), not player guid. Monster `Tile::queryAdd` uses `canpushcreatures`/`canpushitems` defaults **false** until §5.1 `MonsterType` flags parse; magic-field immunity is simplified to block `TILESTATE_MAGICFIELD` unless `FLAG_IGNOREFIELDDAMAGE`. *(`walk.rs`, May 2026)*

17. **Phase D.2 think cadence**: C++ `Game::checkCreatures` passes a **fixed** `EVENT_CREATURE_THINK_INTERVAL` (1000 ms) to `onThink`, not wall-clock drift. Rust mirrors this in `check_creatures`. Flat sweep over all monsters/NPCs in `SlotMap` replaces C++ bucketed `checkCreatureLists[10]` for now (100 ms stagger deferred). Phase D excludes **players** from the think sweep — player ping/idle kick stays separate; C++ runs `Player::onThink` via the same scheduler. First tick only sets `last_creature_check` baseline (no burst). Follow path refresh gating (`walk_update_ticks` / 200 ms threshold / `is_updating_path`) calls `go_to_follow_creature` (D.4). *(`creature_think.rs`, `monster_ai.rs`, May 2026)*

18. **Phase D.3 spawn lifecycle**: C++ `Spawn::startup` uses `internalPlaceCreature` (no `0x6A`); respawn uses `placeCreature` → `sendAddCreature` + `CONST_ME_TELEPORT`. Rust `startup_spawns()` skips appear broadcast; `poll_spawn_respawns` emits `0x6A` per viewer with `check_creature_known`. `findPlayer` checks **spawn block tile** occupants, not zone radius; blocked respawn resets timer (`stall_respawn`). `SpawnManager` returns plans only — `GameWorld` places to avoid borrow conflicts. NPC slots track `current` but `respawns=false`. Monster `<look>` + `<flags>` parsed into `MonsterType` for spawn appearance and AI. *(`spawn.rs`, `spawn_lifecycle.rs`, `monsters.rs`, May 2026)*

19. **Monster DB lookup key is `monsters.xml` index name, not file `name` attr**: C++ `Monsters::loadMonster(file, monsterName)` registers under `monsterName` from the index (`"Red Butterfly"`) while `mType->name` comes from the file (`"Butterfly"` in `red_butterfly.xml`). Rust initially keyed by file attr → spawn `"Red Butterfly"` failed lookup. Fix: parse index `name` + `file` pairs; HashMap key = index name lowercased. *(`monsters.rs`, May 2026)*

20. **Phase D.4 / D.5 monster AI**: C++ `Monster::onThink` calls `Creature::onThink` first — `goToFollowCreature` runs **before** `searchTarget` on the same tick, so a target selected on tick N paths on tick N+1 when `is_updating_path` is set. Rust mirrors order in `monster_on_think` → `creature_on_think` then `monster_native_on_think`. `searchTarget` nearest tie-break uses **Manhattan** sum (`getDistanceX + getDistanceY`), not Chebyshev. Stock TFS idle monsters (`isIdle == true`, empty `targetList`) **do not** random-walk — `getNextStep` returns false immediately (`monster.cpp` ~1226). Active monsters **do** call `addEventWalk()` every think even with an empty queue so `Monster::getNextStep` can random-roam or accept the next flee step (`monster.cpp` ~772). *(`monster_ai.rs`, `creature_think.rs`, May 2026)*

21. **Monster target list uses `Creature::canSee`, not `canSeeCreature` alone**: C++ `Monster::updateTargetList` prunes opponents with `!canSee(creature->getPosition())` — viewport range with z offset (`Map::maxViewportX/Y` = 11), not ghost/invis-only `canSeeCreature`. Rust initially pruned with `can_see_creature` only, so once a player passed within qtree spectator range they stayed in `opponent_ids` forever and the whole cave chased off-screen. Fix: `creature_can_see(viewer_pos, target_pos, 11, 11)` in `monster_creature_visible_to`, `monster_on_creature_found`, and `monster_set_follow_creature` (mirrors `Creature::setFollowCreature`). *(`game_world.rs`, `monster_ai.rs`, May 2026)*

22. **Spawn slot guard + tile path flags**: `process_spawn_request` must skip when `slot.current` is already set — otherwise a respawn tick can place a second monster while the first is still alive (orphans the old creature, looks like “respawn on walk off tile”). Failed placement should `stall_respawn` instead of leaving `current` empty with no timer (tight respawn loop). OTBM `apply_item_tile_flags` must set `TILESTATE_NOFIELDBLOCKPATH` / `IMMOVABLENOFIELDBLOCKPATH` from `blockPathFind` per `item.cpp` `hasProperty`, not only `BLOCKPATH`. *(`spawn_lifecycle.rs`, `map/mod.rs`, May 2026)*

23. **Monster move spectator fan-out**: C++ `Map::moveCreature` notifies all spectators via `onCreatureMove` (`map.cpp` ~264–323) — monsters aggro on enter, drop on leave, and re-path instantly when follow target moves (`creature.cpp` ~619–637). Rust previously deferred follow updates to 1 Hz think only. Also: qtree leaf queries return whole 32×32 leaves — must filter each candidate with `creature_can_see`; `updateTargetList` uses multifloor `getSpectators(..., true)`. Blocked path → `!hasFollowPath` → `getRandomStep` after 1 s (`monster.cpp` ~1224). Login should fan out `onCreatureEnter` to nearby monsters. *(`monster_ai.rs`, `walk.rs`, `login.rs`, May 2026)*

24. **Creature position must update per `Map::moveCreature`, not after `queryDestination` chain**: C++ sets position when `newTile.addThing(&creature)` runs, **before** `onCreatureMove` fan-out (`map.cpp` ~293–324). Rust deferred `set_position(final_pos)` until after the full `queryDestination` loop — during floor-change chains the qtree/tile were at the new z but `creature.position()` still read the old z, so `Monster::onCreatureFound` rejected the player (`creature_can_see` z-band) even when `onCreatureMove` saw `canSee(newPos)`. Fix: `set_position(to)` inside `move_creature_on_map` before `monster_dispatch_creature_move`. **Separate rule:** surface z ≤ 7 cannot see underground z ≥ 8 (`creature.cpp` ~45–56) — rats on z 8 legitimately ignore a player standing on z 7 until they step down onto z 8 (bridge/cave entrance). *(`walk.rs`, `monster_ai.rs`, May 2026)*

25. **Phase D.4 gap closure — walkToSpawn, config radii, dance step**: C++ triggers `walkToSpawn` from `onCreatureLeave` when `targetList` empty and monster outside `walkToSpawnRadius` (axis box via `Position::areInRange`, default 15) — distinct from `deSpawnRadius` (Chebyshev 50) used by `isInSpawnRange`/teleport. Rust wires `monster_maybe_walk_to_spawn` from opponent removal. Config keys: `deSpawnRadius`, `deSpawnRange`, `walkToSpawnRadius`, `removeOnDespawn` → `MonsterWorldConfig` on `GameWorld`. Out-of-range despawn emits `CONST_ME_POFF` then remove or teleport per config. `getDanceStep` ported for flee/chase when follow queue drains (`monster.cpp` ~1244). `onFollowCreatureComplete` reorders `opponent_ids`. **`updateLookDirection` offset:** Rust `offset_x(monster, target)` = `target.x − monster.x` matches C++ `getOffsetX(attackedCreaturePos, pos)` because Rust helper is `to − from` while C++ is `p1 − p2`. Turn/move spectator packets require `creature_fully_sent_by_conn` entry for the wire id. *(`monster_ai.rs`, `monster_distance_step.rs`, `config.rs`, May 2026)*

26. **Phase D.5 follow repath gate**: C++ `Creature::onCreatureMove` only clears `listWalkDir` and schedules `goToFollowCreature` when `hasFollowPath` is true (`creature.cpp` ~619–643). Rust `monster_on_follow_creature_moved` must early-return when `!has_follow_path`; visibility/follow-clear still runs in the caller. `compute_look_toward_target` must use `offset_x(from, target)` not `(target, from)`. Acceptance: `monster_repaths_when_follow_target_moves`. *(`monster_ai.rs`, May 2026)*

27. **Failed monster step must not synchronously repath**: C++ `onWalk` failure only sets `forceUpdateFollowPath`; repath runs from `onThink` / 200 ms follow refresh. 28. **`monster_on_walk_complete` follow loop on empty A* path**: Logs showed `go_to_follow_creature` with `astar_path len=0` → `creature_start_chase_auto_walk` → `on_walk` (`pop_dir_none`) → `monster_on_walk_complete` → repeat until stack overflow (session 963225). 29. **Monster circling / step-back during chase**: Mishandled empty at-goal path plus `getDanceStep` while **approaching** (`dist > target_distance`); Rust also repathed every `on_walk_complete` (C++ does not). Fix: skip dance when `dist > target_distance`; skip no-op repath at goal; empty at-goal path must call `startAutoWalk()`; never disable melee `getDanceStep`. **Dance loop stall**: after the chase queue drains, a failed dance poll set `stopped_without_reschedule` and never re-armed the walk timer — fix with `monster_should_keep_dance_walk_alive` + `schedule_walk_followup_deadline`. First sidestep after arrival may lag one walk cooldown + `staticAttackChance` roll (e.g. rat 90 → ~10% per poll) — matches C++. *(`monster_ai.rs`, `walk.rs`, May 2026)*

19. **Phase A2 version-keyed incoming opcodes**: 7.72 (`gameserver/src/protocolgame.cpp` `parsePacket` ~L466–528) and 10.98 (repo-root `src/protocolgame.cpp`) share the **same byte value** for every common client→server opcode — the only true divergences are the rule-violation flow (772 = `0x9B/0x9C/0x9D` trio; 1098 = single `0xF2`) and the era-exclusive 1098 blocks (shop `0x79–0x7C`, market `0xF4–0xF8`, quest `0xF0/0xF1`, mount `0xD4`, equip `0x77`, wrap `0x8B`, browse/seek `0xCB/0xCC`, VIP-edit `0xDE`, thank-you `0xE7`, get-object-info `0xF3`). So A2 does **not** need a full second opcode table — it needs a support predicate. Implemented `client::is_supported(op, version)` = `SUPPORTED_V1098 − V772_REMOVED + V772_ADDED`, and `parse_game_opcode` rejects opcodes the active era doesn't dispatch before matching. Server-side, only the **self-appear** opcode is genuinely version-keyed (`0x0A` vs `0x17`); exposed as `server::self_appear(version)` reading `ProtocolCaps::self_appear_opcode` (single source of truth, no second constant). All other send opcodes share bytes and differ only in **layout** — that's the codec's job (A5), not the opcode table. `ProtocolVersion` threads from `GameWireConfig` → `forward_game_packets[_xtea]` → `game_command_from_payload` → `parse_game_packet`. 1098 golden bytes unchanged. *(`protocol_opcodes.rs`, `game_parse.rs`, `protocol_game.rs`, May 2026)*

19. **Phase A3 transport caps — cipher offset, not C++ XTEA slack**: The C++ `XTEA_decrypt` length args differ per era — 1098 `xtea::decrypt(buffer, getLength() - 6, …)` (repo-root `src/protocol.cpp`), 772 `xtea::decrypt(buffer, getLength() - 4, …)` (`gameserver/src/protocol.cpp`). That `-6` vs `-4` is **not** a slack to subtract in Rust: it is a byproduct of C++ `NetworkMessage::getLength()` including the inner-size `u16` while `getBufferPosition()` already points past the (era-dependent) header. Our Rust decrypt instead decrypts the **exact** 8-byte-aligned ciphertext region (`xtea_tfs::decrypt`'s loop is `it + 8 <= length`), so the only thing that actually varies is **where the ciphertext starts**. `INITIAL_BUFFER_POSITION` = outer-size(2) + checksum(4 for 1098 / 0 for 772) + inner-size(2): IBP=8 (1098), IBP=4 (772). The framing layer strips the 2-byte outer size and the inner-size `u16` is the first decrypted word, so `cipher_off = IBP - 4` → **4 for 1098, 0 for 772** = exactly the checksum width. First attempt used `IBP - 2` and every frame failed "game body too short"; the `ProtocolCaps.xtea_length_slack` field (4/6) is therefore informational only — the real driver is `adler_checksum` / `initial_buffer_position`. 1098 output stays byte-identical (offset 4 == the old hardcoded `body[4..]`). Pre-login `0x1F` challenge gated by `caps.prelogin_challenge`; `send_game_challenge` now returns `Option<GameChallenge>` and the echo is verified only when `Some`. *(`protocol_game.rs`, `game_challenge.rs`, `server.rs`, May 2026)*

14. **772 login is account-*number* + inline credentials; no session key (A4)**: The 7.72 "account number" is `accounts.id`, not a separate column — both `loginserverAuthentication` and `gameworldAuthentication` in `gameserver/src/iologindata.cpp` query `SELECT … FROM accounts WHERE id = {accno}`. The game-port first packet (`gameserver/src/protocolgame.cpp onRecvFirstMessage`) is `skipBytes(1)` gm flag + `u32 accountNumber` + `string character` + `string password` + OTCv8 probe — no `acc\npass\ntoken\ntime` session-key string and no `0x1F` challenge echo (both 1098-only, gated by `caps.session_key_login` / `caps.prelogin_challenge`). The login-port char list (`getCharacterList`) writes per-char `name + serverName + u32 inet_addr(ip) + u16 port` and premium as a single `u16` days value (`u16::MAX` for free-premium, else `floor(secsLeft/86400)+1`); `disconnectClient` uses opcode `0x0A` vs 1098 `0x0B`. Modeled identity as `LoginIdentity::{AccountName(String), AccountNumber(u32)}` filled from `ProtocolCaps`, and split the credential parse into pure `parse_{game,login}_credentials` fns so both eras are unit-testable without a socket. `inet_addr` on a little-endian host yields octets in `a.b.c.d` order as a LE `u32`, which `add<uint32_t>` then writes verbatim — so we store `u32::from_le_bytes([a,b,c,d])`. The exact 772 first-packet *framing* offsets (proto-id byte `0x0A`/`0x01` + prelude widths) are tagged `// PROTOCOL:` and must be confirmed against a live 7.72 capture in A6. *(`game_first_packet.rs`, `protocol_login_out.rs`, `account.rs`, May 2026)*

**Phase A5/A6 — `Codec772` layouts + neutral-struct widening**: The 772 wire is mostly *narrower* than 1098, but two neutral wire structs had to **widen** (design §9.5 — never pre-narrow in core, let the codec narrow):
- `AddCreatureWire.speed_half → step_speed`: 772 `AddCreature` writes the **full** `getStepSpeed()` (`gameserver/src/protocolgame.cpp ~L2079`), 1098 writes `getStepSpeed()/2` (`src/protocolgame.cpp ~L3210`). Core (`login_out.rs`) now fills the full value; `Codec1098::write_add_creature` halves it, `Codec772` writes it raw. Keeping the half in core would have lost the low bit for 772.
- New `ContainerOpenWire` (max-width `sendContainer`).
